# Техническое задание (ТЗ)

**Проект:** WikiLive  
**Версия документа:** 1.0

## Назначение документа

Этот документ фиксирует подробное техническое задание на хакатонный MVP проекта WikiLive на основе исходной переписки по задаче, с уточнениями по стеку, ограничениям, составу команды, объёму работ и критериям готовности. Исходные договорённости взяты из загруженного файла с перепиской; архитектура дополнительно скорректирована по последнему уточнению: убран отдельный JS/Node-контейнер на бэкенде, real-time и WebSocket-часть должны быть реализованы без него, а на бэкенде допускаются только два языка — Java и Go.

## 1. Краткое описание проекта

### 1.1. Рабочее название
WikiLive (на текущий момент — рабочее название, не обязательно финальное).

### 1.2. Идея проекта
WikiLive — это вики/документный инструмент с упором на живые таблицы внутри текстового контента, где текст и таблица существуют не как отдельные сущности, а как единый документ. Ключевое отличие от классических wiki-платформ — поддержка совместного редактирования в реальном времени, встроенных таблиц и современного UX редактирования.

### 1.3. Что именно делаем на хакатоне
На хакатоне не делается «полноценный клон Notion». Вместо этого делается сфокусированный MVP-модуль:

> Текстовая wiki-страница, внутри которой можно вставлять и редактировать живую таблицу в реальном времени несколькими пользователями.

1. Пользователь авторизуется;
2. Открывает страницу;
3. Редактирует rich-text-контент;
4. Вставляет в текст таблицу;
5. Другой пользователь открывает ту же страницу;
6. Оба пользователя одновременно видят изменения текста/таблицы в реальном времени.

## 2. Контекст и продуктовая мотивация

### 2.1. Проблема
Существуют два крайних подхода:
- Классические wiki-системы с устаревшим UX и слабой совместной работой;
- Современные инструменты вроде Notion / Coda / Docmost / Google Docs, которые либо перегружены, либо решают более широкий класс задач.

Промежуток, который должен занять WikiLive:
- Простой и понятный wiki-like опыт;
- Современное редактирование;
- Встроенные таблицы как часть документа;
- Совместная работа в реальном времени.

### 2.2. Почему это актуально
Для команд, работающих в гибридном режиме, важны:
- Единое пространство для знаний;
- Живое редактирование, а не только хранение статичных документов;
- Быстрый способ собирать структурированную информацию прямо в тексте;
- Отсутствие переключения между редактором документа и отдельным табличным инструментом.

### 2.3. Рыночные ориентиры
Проект ориентируется на лучшие элементы следующих классов решений:
- MediaWiki — как база идеи wiki-страниц;
- Google Docs — по части совместного редактирования и курсоров;
- Notion — по UX встроенных блоков и inline-встраивания;
- Coda / Docmost — по идее "живых" объектов внутри документа.

При этом цель MVP — не догнать все функции этих продуктов, а доказать основной сценарий ценности.

## 3. Цели, границы и ожидания от MVP

### 3.1. Цель MVP
Собрать за хакатон демонстрационно убедительный продукт, в котором работают все следующие свойства одновременно:
- Есть авторизация;
- Есть wiki-страницы;
- Есть текстовый редактор страницы;
- В документ можно встроить таблицу;
- Таблица редактируется inline, без перехода в отдельный режим;
- 2+ пользователя видят изменения в реальном времени;
- Система разворачивается контейнерно и годится для демо.

### 3.2. Главный пользовательский результат
После демо должно быть очевидно:
- Продукт решает сценарий совместного ведения знаний;
- Таблицы не «прикреплены отдельно», а естественно живут внутри текста;
- Архитектура уже пригодна для дальнейшего роста.

### 3.3. Что считается успехом на хакатоне
Успехом считается не максимальное число фич, а:
- Стабильный end-to-end сценарий;
- Понятная и демонстрируемая ценность;
- Отсутствие архитектурных решений, которые придётся полностью выкидывать после хакатона;
- Нормальная UX-демонстрация без развала real-time части.

## 4. Зафиксированные ограничения и принятые решения

### 4.1. Команда
- Илья — DevOps;
- Кирилл — Frontend + PM;
- Миша — Backend Java;
- Равиль — Backend Go.

UX/UI-дизайн — без выделенного полноценного дизайнера, дизайн закрывается командой по совместительству.

### 4.2. Предпочтительный стек

**Frontend:**
- React;
- TanStack Start;
- File-based routing;
- Tailwind CSS;
- Rich-text editor: **BlockNote** как базовый редактор.

**Backend:**
На бэкенде допускаются только два языка: **Java** и **Go**.

> **Важное изменение относительно исходного обсуждения:**  
> Из целевой схемы убирается отдельный JS/Node-сервис для real-time-синхронизации. Вместо этого real-time и WebSocket-слой должны быть реализованы на Go (предпочтительно) или в рамках допустимой двухъязычной backend-схемы Java+Go без третьего backend-языка.

### 4.3. Базы и хранилища
- **PostgreSQL** — пользователи, роли, доступы, auth-данные и системные сущности;
- **MongoDB** — страницы, структурированный контент, метаданные документа, при необходимости снапшоты/операционные структуры редактора;
- **S3-совместимое хранилище** (например, MinIO) — медиа и вложения.

### 4.4. Деплой
- Обязателен Docker;
- Обязателен docker-compose;
- Для демо может использоваться любой удобный вариант, если он способен поднять всю систему;
- Допускается Vercel для фронтенда, если это не ломает общую сборку и демо-сценарий.

### 4.5. Режим работы приложения
- Только онлайн;
- Офлайн-first / PWA для MVP не требуется.

### 4.6. Авторизация
Для MVP допустимо:
- JWT-based auth;
- Без обязательного OAuth/SSO;
- Базовый сценарий email + password.

### 4.7. Структура страниц
- Страницы плоские, без вложенности страница-в-странице.

### 4.8. Роли
Базовые роли на MVP:
- Владелец;
- Редактор;
- Читатель.

### 4.9. Таблицы в MVP
На первом этапе таблицы только базовые:
- Строки;
- Столбцы;
- Ячейки;
- Редактирование содержимого ячеек.

На MVP **не обязательны**:
- Формулы;
- Фильтры;
- Сортировки;
- Представления;
- Связи между таблицами.

### 4.10. AI-функции
AI-компонент в MVP не нужен.

### 4.11. Репозиторий
- Монорепозиторий;
- Отдельные директории по приложениям/сервисам.

### 5.1. In Scope (обязательно в рамках хакатона)

4. Вложенные страницы.
5. Сложные шаблоны документов.
6. Формулы в таблицах.
7. Фильтры и сортировки таблиц.
8. Комментарии и обсуждения.

13. SSO / OAuth.

### 5.3. Stretch Goals (делаем только если есть запас времени)

2. Live-курсоры и выделения.

7. Базовые блоки callout / heading / checklist, если BlockNote это даёт быстро.

## 6. Основные пользовательские сценарии

### 6.1. Сценарий 1. Вход в систему
**Актор:** пользователь  
**Шаги:**
1. Пользователь открывает приложение;
2. Переходит на экран входа/регистрации;
3. Создаёт аккаунт или логинится;
4. Получает доступ к списку страниц или стартовой странице.

**Ожидаемый результат:** пользователь аутентифицирован и может работать в системе в рамках своей роли.

### 6.2. Сценарий 2. Создание страницы
**Актор:** владелец/редактор  
**Шаги:**
1. Пользователь нажимает «Создать страницу»;
2. Указывает название;
3. Открывается редактор новой страницы.

**Ожидаемый результат:** страница создана и доступна для редактирования.

### 6.3. Сценарий 3. Редактирование текста
**Актор:** владелец/редактор  
**Шаги:**
1. Пользователь открывает страницу;
2. Вводит обычный текст;
3. Добавляет заголовки/абзацы/базовые блоки;
4. Изменения сохраняются и отражаются другим участникам.

**Ожидаемый результат:** текст страницы изменён и сохранён.

### 6.4. Сценарий 4. Вставка таблицы в документ
**Актор:** владелец/редактор  
**Шаги:**
1. Пользователь ставит курсор в нужное место документа;
2. Выбирает действие вставки таблицы;
3. Создаётся таблица с начальными размерами;
4. Таблица отображается как часть документа.

**Ожидаемый результат:** таблица встроена внутрь контента и не является отдельным экраном/режимом.

### 6.5. Сценарий 5. Совместное редактирование
**Акторы:** два и более пользователя  
**Шаги:**
1. Пользователь А открывает страницу;
2. Пользователь В открывает ту же страницу;
3. Пользователь А меняет текст или ячейку таблицы;
4. Пользователь В видит изменения почти мгновенно;
5. Пользователь В редактирует другую часть документа;
6. Пользователь А видит эти изменения без ручного обновления страницы.

**Ожидаемый результат:** обе стороны наблюдают синхронное состояние страницы в реальном времени.

### 6.6. Сценарий 6. Ограничение доступа
**Актор:** читатель  
**Шаги:**
1. Пользователь с ролью читателя открывает страницу;
2. Интерфейс отображает содержимое;
3. Редактирование недоступно.

**Ожидаемый результат:** доступы работают, пользователь не может изменить документ.

## 7. Функциональные требования

### 7.1. Модуль авторизации и пользователей
Система должна:
1. Поддерживать регистрацию пользователя по email/password;
2. Поддерживать вход по email/password;
3. Выдавать JWT для авторизованных сессий;
4. Валидировать JWT на backend-уровне;
5. Различать роли пользователя;
6. Ограничивать операции в зависимости от роли.

**Минимальные сущности:** User, Role, PagePermission / AccessRule.

### 7.2. Модуль страниц
Система должна:
1. Позволять создавать страницу;
2. Хранить заголовок страницы;
3. Хранить тело страницы;
4. Открывать страницу по идентификатору;
5. Отображать страницу в редакторе;
6. Сохранять состояние страницы.

**Минимальные поля страницы:** id, title, content/document state, ownerId, timestamps, access settings.

### 7.3. Модуль редактора
Система должна:
1. Отображать rich-text редактор;
2. Поддерживать базовые текстовые блоки;
3. Позволять вставлять таблицу внутрь документа;
4. Позволять редактировать содержимое таблицы inline;
5. Обеспечивать согласованное сохранение структуры документа.

### 7.4. Модуль таблиц
Система должна:
1. Создавать таблицу внутри документа;
2. Поддерживать базовую сетку ячеек;
3. Позволять редактировать значение ячейки;
4. Позволять добавлять/удалять строки и столбцы, если команда успевает в MVP;
5. Синхронизировать изменения таблицы между участниками в real-time.

> **Важно:** Таблица — это часть документа, а не отдельный внешний объект с отдельным UI-флоу.

### 7.5. Модуль real-time синхронизации
Система должна:
1. Устанавливать WebSocket-соединение между клиентом и backend;
2. Подписывать клиента на изменения по конкретной странице;
3. Рассылать изменения участникам страницы;
4. Обеспечивать near real-time обновление;
5. Корректно переживать одновременные изменения нескольких пользователей;
6. Не требовать ручного refresh страницы.

**Архитектурное уточнение:**  
Для MVP допускается упрощённая модель real-time без production-grade реализации всех edge-case конфликтов, но с обязательной демонстрацией:
- Синхронного отображения изменений;
- Отсутствия явного развала документа при совместной работе;
- Предсказуемого поведения в типовых happy-path сценариях.

### 7.6. Модуль ролей и доступа
Система должна поддерживать роли: владелец, редактор, читатель.

**Права по ролям:**
- **Владелец:** полный доступ, может редактировать, может назначать роли, может открывать/закрывать доступ.
- **Редактор:** может читать, может редактировать.
- **Читатель:** может только просматривать.

### 7.7. Модуль API
Система должна иметь backend API для:
- Регистрации/логина;
- Получения профиля текущего пользователя;
- Создания страницы;
- Чтения страницы;
- Обновления метаданных страницы;
- Изменения прав доступа;
- Загрузки/получения состояния документа;
- Подключения к live-каналу страницы.

## 8. Нефункциональные требования

### 8.1. Производительность
Для MVP достаточно следующих ориентиров:
- Страница открывается без заметной деградации UX на типовом контенте;
- Live-обновления приходят визуально быстро и достаточно для демо;
- Работа 2–5 одновременных пользователей на одной странице не ломает сценарий.

### 8.2. Надежность
- Сервисы должны подниматься одной командой через docker-compose;
- После перезапуска backend ранее сохранённые страницы должны оставаться доступными;
- Ошибки одного пользовательского действия не должны полностью ломать весь документ.

### 8.3. Безопасность
- Закрытые маршруты должны требовать валидный токен;
- Действия редактирования должны проверять права;
- Не должно быть очевидной возможности анонимно редактировать чужую страницу;
- Пароли не должны храниться в открытом виде.

### 8.4. Поддерживаемость
- Структура проекта должна быть прозрачной;
- Backend-ответственности должны быть разделены между Java и Go;
- Монорепозиторий должен иметь понятную структуру директорий;
- Переменные окружения должны быть документированы.

### 8.5. Демонстрационная пригодность
- Приложение должно быть можно поднять локально или на стенде без ручной магии;
- Демо-сценарий должен быть повторяемым;
- Подготовка окружения перед показом должна быть минимальной.

## 9. Пересобранная целевая архитектура MVP

### 9.1. Архитектурный принцип
Архитектура должна сохранить разделение ответственности, но быть достаточно простой, чтобы успеть на хакатоне.

### 9.2. Компоненты

1.  **Frontend (React/TanStack Start)**
    - UI, роутинг, редактор, отображение документа, работа с auth, работа с WebSocket-каналом, UX inline-таблиц.

2.  **Java backend service**
    - Auth, security, users, roles/permissions, часть orchestration-логики, возможно API gateway/application facade.

3.  **Go backend service**
    - Страницы, документные модели, операции над контентом, WebSocket-соединения, real-time sync, доставка live-обновлений участникам страницы, при необходимости — API для page/document state.

4.  **PostgreSQL**
    - Хранение auth-данных, ролей, прав доступа, системных связей.

5.  **MongoDB**
    - Хранение структуры страниц, редакторного состояния, контента документа, таблиц внутри документа, опционально снапшотов и событий.

6.  **MinIO / S3-compatible storage**
    - Хранение файлов и медиа, если команда успеет включить вложения.

7.  **Reverse proxy / API gateway**
    - Нужен единый entry point. Это инфраструктурный компонент, а не третий backend-язык.

### 9.3. Что убрано из старой схемы
Убирается отдельный Node/JS WS/CRDT-сервис. **Это обязательное изменение.**

### 9.4. Как теперь реализовать real-time
- Клиент открывает WebSocket к Go-сервису;
- Go-сервис держит комнаты/каналы по pageId;
- Клиент отправляет изменения документа/таблицы;
- Go-сервис валидирует контекст и рассылает обновления подписанным участникам;
- Устойчивость к конфликтам обеспечивается на уровне упрощённого протокола синхронизации, достаточного для демо.

### 9.5. Что важно не переусложнить
На хакатоне нельзя провалиться в бесконечный ресёрч CRDT/OT. Если полноценная CRDT-реализация не помещается по срокам, допускается:
- Упрощённый live-sync для happy-path;
- Блокировка/дебаунс/батчинг на уровне отдельных операций;
- Фиксированный набор редактируемых операций.

## 10. Распределение ответственности между сервисами

### 10.1. Java service
**Обязанности:** регистрация, логин, выдача/валидация JWT, роли и права, информация о пользователях, управление доступом к страницам, защитный слой для API.  
**Не должен брать на себя:** WebSocket live-коммуникацию, реалтайм-комнаты, основную логику синхронизации документа (если это уже реализуется в Go).

### 10.2. Go service
**Обязанности:** CRUD/операции страниц, документный state, таблицы внутри документа, WebSocket real-time, рассылка live-обновлений, обработка конкурентных изменений в рамках MVP-логики.  
**Не должен брать на себя:** сложный auth lifecycle (если он уже централизован в Java), дублирование role-management.

### 10.3. Граница между сервисами
- Java отвечает на вопрос: *кто пользователь и что ему можно*;
- Go отвечает на вопрос: *какой документ он видит и как он меняется в реальном времени*.

## 13. UI/UX требования для MVP

### 13.1. Основной UX-принцип
Пользователь не должен ощущать, что таблица — это внешний модуль. Она должна ощущаться как органичная часть страницы.

### 13.2. Минимальные экраны
1. Экран логина/регистрации.
2. Экран списка страниц или стартовый экран.
3. Экран просмотра/редактирования страницы.
4. Базовый UI управления доступом (можно минимально, если успеете).

### 13.3. Требования к экрану страницы
- Название страницы;
- Редактор контента;
- Кнопка/механизм вставки таблицы;
- Визуально понятный режим редактирования;
- Индикатор сохранения или live-статуса, если успеете.

### 13.4. Требования к таблице
- Быть читаемой внутри документа;
- Позволять кликнуть в ячейку и редактировать её;
- Не ломать поток редактирования страницы;
- Выглядеть как единый элемент с документом.

### 13.5. UX, который не обязателен, но желателен
- Индикатор присутствия пользователей;
- Подсказка, что страница обновляется в реальном времени;
- Минимальный empty-state;
- Минимальная обработка ошибок соединения.

## 14. API и интеграционные контракты (на уровне требований)

### 14.1. Auth API
- Регистрация;
- Логин;
- Получение текущего пользователя;
- Logout на клиентском уровне.

### 14.2. Page API
- Создать страницу;
- Получить страницу;
- Обновить title;
- Обновить content state;
- Получить список доступных страниц пользователя.

### 14.3. Access API
- Назначить роль пользователю на страницу;
- Получить список пользователей с ролями по странице.

### 14.4. Live API / WS protocol
События/сообщения для:
- Подключения к странице;
- Подписки на page room;
- Отправки изменений документа;
- Получения изменений документа;
- Уведомления об ошибке/разрыве соединения.

## 15. Сценарий демо

### 15.1. Обязательный демо-сценарий
1. Показать логин пользователя А.
2. Открыть существующую или создать новую страницу.
3. Написать текст на странице.
4. Вставить таблицу в середину документа.
5. Заполнить несколько ячеек.
6. Открыть ту же страницу под пользователем В.
7. Изменить текст/ячейки с одного клиента.
8. Показать live-обновления на втором клиенте.
9. Показать, что читатель не может редактировать.

### 15.2. Что особенно важно для впечатления от демо
- Видимая синхронность;
- Отсутствие ручного refresh;
- Таблица действительно встроена в текст;
- Простая и понятная логика ролей.

## 16. Риски проекта

### 16.1. Главный риск
Самый опасный риск — переусложнить real-time слой и не собрать работающее демо.

### 16.2. Риски по направлениям
- **Product risk:** распыление на слишком широкий набор фич, попытка строить Notion целиком.
- **Frontend risk:** слишком много времени уйдёт на кастомизацию редактора, таблица будет выглядеть отдельным нестабильным компонентом.
- **Backend risk:** переусложнение протокола синхронизации, слишком тонкая граница между Java и Go, конфликт ответственности сервисов.
- **DevOps risk:** сложный деплой и несобираемое окружение, проблемы межсервисного соединения на демо, несогласованные переменные окружения.

### 16.3. Меры снижения рисков
1. Заморозить scope MVP в первый день.
2. Не делать CRDT-исследование бесконечно, если можно показать simpler live-sync.
3. Рано собрать вертикальный slice: login → page → editor → table → ws.
4. Docker-сборку проверить заранее.
5. Демо-сценарий прогонять до финальной презентации несколько раз.

## 17. Приоритеты реализации

**Р0 — обязательно:**
- Авторизация;
- Страница;
- Rich-text;
- Вставка таблицы;
- Inline-редактирование таблицы;
- Live-sync хотя бы для ключевых операций;
- Роли owner/editor/reader;
- Docker-compose;
- Стабильный демо-сценарий.

**Р1 — желательно:**
- Список страниц;
- Управление доступом через UI;
- Presence-индикаторы;
- Несколько участников;
- Базовая обработка reconnection.

**Р2 — только если останется время:**
- Live-курсоры;
- Снапшоты/история;
- Загрузка медиа;
- Дополнительные блоки редактора;
- Косметический polish.

## 18. Definition of Ready (DoR)

Задача/подзадача считается готовой к началу разработки, если:
1. Понятен пользовательский сценарий;
2. Понятно, к какому сервису относится задача;
3. Понятен ожидаемый результат;
4. Понятен минимум данных/контракта;
5. Зафиксировано, входит ли задача в Р0/Р1/Р2;
6. Нет противоречия с главным ограничением: на backend только Java и Go;
7. Не требуется писать код «на удачу» без согласования API/данных.

## 19. Definition of Done (DoD) — общий

Задача считается завершённой, если одновременно выполнено всё нижеперечисленное:
1. Реализован заявленный функциональный результат;
2. Результат интегрирован в общий сценарий, а не существует изолированно;
3. Нет блокирующих ошибок в happy-path;
4. Изменения задокументированы на уровне, достаточном для команды;
5. Функциональность проверена вручную;
6. Сервис/экран/модуль собирается в целевом окружении;
7. Задача не нарушает архитектурные ограничения текущего MVP.

## 20. Предметные DoD по ключевым блокам

### 20.1. DoD — авторизация
- Пользователь может зарегистрироваться;
- Пользователь может залогиниться;
- Клиент получает и использует JWT;
- Защищённые маршруты недоступны без токена;
- Пароль не хранится в открытом виде;
- Happy-path входа проверен вручную.

### 20.2. DoD — страница
- Страницу можно создать;
- Страницу можно открыть по id;
- У страницы сохраняется title;
- У страницы сохраняется content state;
- После перезагрузки страницы данные не теряются;
- Владелец/редактор может редактировать, читатель — нет.

### 20.3. DoD — rich-text редактор
- Редактор отображается корректно;
- Пользователь может вводить и сохранять текст;
- Базовые блоки работают без визуальных поломок;
- Редактор интегрирован со структурой страницы;
- Текст не пропадает при повторном открытии страницы.

### 20.4. DoD — таблица внутри документа
- Таблицу можно вставить внутрь документа;
- Таблица визуально отображается как часть документа;
- Ячейки можно редактировать;
- Изменение ячеек сохраняется;
- Таблица не выбрасывает пользователя в отдельный экран/режим;
- Таблица не ломает документную структуру.

### 20.5. DoD — real-time синхронизация
- Два клиента могут одновременно открыть одну страницу;
- Изменение на одном клиенте отображается на другом без refresh;
- WebSocket-соединение поднимается стабильно;
- Синхронизация работает хотя бы в типовых happy-path действиях;
- Одновременное редактирование не приводит к очевидному разрушению документа в демонстрационном сценарии.

### 20.6. DoD — роли и доступы
- В системе есть роли owner/editor/reader;
- Права реально влияют на поведение UI и API;
- Читатель не может редактировать;
- Редактор может редактировать;
- Владелец имеет полный доступ к странице.

### 20.7. DoD — docker/deploy
- Проект поднимается через docker-compose;
- Все нужные сервисы стартуют;
- Фронтенд доступен;
- Backend-сервисы доступны;
- Базы данных подключаются;
- Команда запуска описана в документации;
- Демо можно повторить в собранном окружении.

## 21. QA-чеклист для полной проверки такси

### 21.1. Проверка авторизации
- Регистрация;
- Вход;
- Обработка невалидных данных;
- Доступ к защищённым страницам после логина;
- Невозможность редактирования без логина.

### 21.2. Проверка страниц
- Создание;
- Повторное открытие;
- Сохранение title;
- Сохранение текста;
- Сохранность после перезагрузки.

### 21.3. Проверка таблиц
- Вставка внутрь текста;
- Корректный рендер;
- Редактирование ячейки;
- Сохранение значения;
- Сохранность как части документа после перезагрузки.

### 21.4. Проверка real-time
- Одновременное открытие страницы двумя пользователями;
- Видимость изменений текста;
- Видимость изменений таблицы;
- Отсутствие необходимости ручного refresh;
- Целостность документа при последовательных изменениях.

### 21.5. Проверка ролей
- Owner может редактировать;
- Editor может редактировать;
- Reader не может редактировать;
- Ограничения работают на UI и backend.

### 21.6. Проверка деплоя
- docker-compose поднимает проект;
- Переменные окружения заданы;
- Сервисы соединяются между собой;
- База инициализируется;
- Демо можно показать на поднятом стенде.

## 22. Критерии приёмки MVP

MVP принимается, если выполнены все обязательные условия:

1. Есть рабочая авторизация;
2. Есть рабочая wiki-страница;
3. В странице работает rich-text ввод;
4. В страницу можно встроить таблицу;
5. Таблица редактируется inline;
6. Два пользователя видят изменения в реальном времени;
7. Есть минимум ролей и ограничений доступа;
8. Проект собирается и запускается контейнерно;
9. Команда может без костылей показать end-to-end демо.

> **Критично:** Если отсутствует хотя бы один из пунктов 4–6, ценность именно идеи WikiLive считается нераскрытой.
